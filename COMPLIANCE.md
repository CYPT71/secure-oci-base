# Automated control evidence matrix

This matrix records technical evidence generated by CI. It is not a claim of certification or organizational compliance.

| Framework | Automated technical evidence | Manual or organizational responsibility |
|---|---|---|
| OCI Image Specification | Generated layout, descriptor, digest, blob, and filesystem checks in `oci-validation` | Runtime acceptance testing for every target platform |
| ISO/IEC 27001 / 27002 | Source review gates, reproducible artifact checks, digest evidence, secret and vulnerability scanning | ISMS scope, risk treatment, asset ownership, access review, audit program |
| NIST SSDF | Secure build, test, code scanning, release checksum evidence | Supplier governance, vulnerability response process |
| NIST SP 800-190 / CIS | Non-root image metadata and documented runtime hardening | Cluster admission policy, capability dropping, read-only root filesystem enforcement |
| OWASP | Input validation, dependency/vulnerability and secret scanning | Threat modeling and operational monitoring |
| SLSA / OpenSSF | Protected CI build records, immutable digest evidence, provenance-capable release permissions | Trusted builders, branch protection, review policy, signing-key governance |

Artifacts uploaded by the workflows are evidence for a particular run and must be retained according to the organization’s audit and retention policy. Security findings, publication failures, or failed quality gates reject the relevant pipeline stage; they require investigation rather than a bypass.
